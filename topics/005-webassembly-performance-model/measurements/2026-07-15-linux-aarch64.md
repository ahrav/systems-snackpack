# Linux AArch64 measurement

## Host and build

- Host: `dev-dsk-ahrav-2b-7dc7bd93.us-west-2.amazon.com`.
- Runtime probe: `aarch64`; EC2 `c7g.16xlarge`; Arm MIDR
  `0x00000000411fd401`, model 1, stepping r1p1; 64 available CPUs.
- OS: Amazon Linux 2023; kernel
  `6.12.94-123.176.amzn2023.aarch64`.
- Toolchain: GCC 11.5.0; Python 3.9.25; rustc/Cargo 1.95.0; LLVM 22.1.2.
- Engine: official Wasmtime 46.0.1 (`823d1b8f2`, 2026-06-24), Cranelift,
  optimization `speed`, parallel compilation off, no target override.
- Target evidence: `asimd`, SVE, atomics, AES/PMULL/SHA, CRC32, i8mm, and
  bf16 appear in the recorded CPU flags. The module uses only core scalar
  `i64`, control flow, a direct call, and one typed import.
- Build: `gcc -O2 -std=c11 -Wall -Wextra -Werror` with the Wasmtime C API.
- The manifest records the WAT, C embedder, and process-runner SHA-256
  values of the sources that produced this evidence, taken at evidence
  commit `3fe13cb` (pinned in the workspace-gate log). The harness in this
  topic has been revised since that commit; `boundary.wat`, the measured
  arithmetic, and the timed phases are unchanged.
- CPU-frequency governor and boost controls were unavailable. `perf` was not
  installed. Measured child processes ran on CPU 0 through `taskset`.

## Measurement boundary

Both exports receive one typed C-to-Wasm call. `guest_loop` applies 10 million
dependency-chained wrapping multiply-add steps through a direct Wasm call.
`host_loop` applies the same arithmetic through 10 million typed C callbacks.
`CLOCK_MONOTONIC_RAW` surrounds each whole export call.

Each fresh process compiles and instantiates once, performs three excluded
100,000-iteration warmups per path, and measures one guest/host pair. Two
whole-process warmups precede 12 retained processes. Order alternates six
`GH` and six `HG` pairs. The process is the experimental unit; iterations are
subsamples.

| Added callback path | 71.899544 ns/step | 1.164077 | 70.654546–75.834087 | 71.075934–75.039677 |
| Callback/direct ratio | 45.772363× | 0.421456 | 44.166363–48.285939× | 45.435591–48.129409× |

The guest-first ratio median was `45.874938×`; the host-first median was
`45.772363×`. Order balancing remains necessary.

For 12 process values, `[X_(3), X_(10)]` has exact 96.1426% coverage for the
population median under independent, identically distributed, continuous
samples. The runner alternates order deterministically, so the pooled sample
mixes the two order strata; under an order effect the stated coverage holds
only within that iid idealization, and the per-order medians above quantify
the observed order-stratum difference. This interval covers fresh-process
variation in this host/build/time block only.

## Startup and correctness

| Phase | Median |
|---|---:|
| WAT load | 48.264 µs |
| WAT decode | 174.753 µs |
| Engine create | 69.792 µs |
| Validate | 90.647 µs |
| Module compile | 1.254750 ms |
| Store and typed import setup | 29.370 µs |
| Instantiate | 9.046 µs |
| Export lookup | 5.709 µs |
| `main` entry to ready | 1.716780 ms |
| Internal warmup | 22.880430 ms |
| External whole process | 787.436896 ms |

`wasmtime_module_new` validates the module bytes again internally, so the
module-compile row includes a second validation pass and `main` entry to
ready contains both passes. The validate row measures the standalone pass
only; the rows repeat work and do not sum to a distinct-phase readiness path.

All 12 measured processes returned
`guest_result == host_result == expected == 0x85463ddc01d7d46f` and exactly
10,000,000 callbacks. The external process time also includes `taskset`,
`exec`, dynamic loading, runner bookkeeping, native correctness work, output,
cleanup, and teardown. It is not pure Wasm startup.

## Observed code and inference

The serialized-module disassembly shows `madd x2, x4, x11, x12` in
`guest_step`, a direct `bl` in `guest_loop`, and an indirect `blr x20` after
loading callback and context fields in `host_loop`. Both loops retain a
stack-limit trap check.

Elapsed times and instructions are measured or observed. The paired difference
is derived. Attributing that difference to the typed import path is an
inference that includes trampoline, runtime-state, value-handling, indirect
transfer, and host callback work. It is not a universal AArch64 or Wasm cost.
