## Process-level experiment

This loop starts nine benchmark processes:

```bash
for run in $(seq 1 9); do
  taskset -c 0 cargo bench --quiet \
    -p systems-snackpack-topic-004 --bench division_chain
done
```

Each process runs six warm-up pairs and 21 measured samples per path. Each
sample contains 10,000,000 loop-carried iterations. The program alternates path
order within each process and reports each path's median whole-kernel time.

|---:|---:|---:|---:|
| 1 | 32,504,951 ns | 51,478,598 ns | 1.583716 |
| 2 | 32,509,250 ns | 51,461,927 ns | 1.582993 |
| 3 | 32,507,941 ns | 51,449,378 ns | 1.582671 |
| 4 | 32,506,250 ns | 51,457,587 ns | 1.583006 |
| 5 | 32,509,819 ns | 51,451,846 ns | 1.582656 |
| 6 | 32,508,029 ns | 51,468,748 ns | 1.583263 |
| 7 | 32,506,831 ns | 51,462,967 ns | 1.583143 |
| 8 | 32,513,622 ns | 51,461,668 ns | 1.582773 |
| 9 | 32,512,262 ns | 51,466,618 ns | 1.582991 |

The median of the nine constant-path process medians is 32,508,029 ns, or
3.251 ns per iteration. The constant-path process medians range from
32,504,951 to 32,513,622 ns. The median of the nine runtime-path process
medians is 51,461,927 ns, or 5.146 ns per iteration. The runtime-path process
medians range from 51,449,378 to 51,478,598 ns.

The ratio column recomputes each value from the raw nanosecond totals rather
than the three-decimal ratio printed by the benchmark. The median paired ratio
is `1.582993x`. If the nine paired ratios are independent, identically
distributed samples from one continuous process-run distribution, the second
and eighth ordered ratios form an exact 96.1% sign-based confidence interval
for that distribution's median:
`[1.582671, 1.583263]`. The interval quantifies sampling uncertainty for this
host and time block. It is not a range for future runs and does not cover build,
host, or fleet variation.

## Release codegen observation

Command:

```bash
cargo rustc --quiet -p systems-snackpack-topic-004 \
  --release --lib -- --emit=asm
```

LLVM unrolled both loops by two iterations. The runtime loop checked whether
the numerator or divisor had nonzero upper 32 bits. It used `divq` when either
did and `divl` when both values fit in 32 bits. Its repeated shape included:

```asm
imulq   %r8, %rax
addq    %rdi, %rax
movq    %rax, %rdx
orq     %rsi, %rdx
shrq    $32, %rdx
je      .LBB0_4
xorl    %edx, %edx
divq    %rsi
```

The constant loop used multiply-high reduction:

```asm
imulq   %r9, %rdi
addq    %rcx, %rdi
movq    %rdi, %rax
mulq    %r8
subq    %rdx, %rdi
shrq    %rdi
addq    %rdx, %rdi
shrq    $2, %rdi
```

The runtime measurement includes the upper-half check, its branch, and the
selected divide form for each iteration. The experiment did not record the
branch-outcome distribution and does not isolate 64-bit divide latency. These
sequences establish lowering for rustc 1.93.1 and this target, not every x86-64
CPU.

## Measurement boundary

The two clock reads span one non-inlined kernel call. The interval includes the
call and return, multiply-add, division or replacement sequence, dependency
chain, and loop control. Argument and result `black_box` calls, process startup,
warmup, validation, sorting, and output occur outside the interval.

The elapsed times and paired ratios are measured. The instruction sequences are
observed codegen. The constant and runtime paths differ in emitted division
lowering, but the timer includes every instruction in each loop. These data do
not assign the ratio to `divq`, `divl`, `mulq`, or any single core resource.
