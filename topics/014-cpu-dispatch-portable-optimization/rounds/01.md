# Round 01: legality is not profitability

## Question

How should a portable binary select specialized CPU code without confusing
instruction legality with workload profitability?

## Finding

Use a conservative deployment baseline, isolate each specialized implementation
behind its complete runtime feature contract, and keep selection policy outside
the unsafe call. Hoist that decision to a coarse boundary. A feature bit says
that a path may execute; only measurements on the relevant host and workload
show whether it should execute.

## Artifact

The crate provides a scalar oracle, AVX2 and AArch64 Advanced SIMD variants,
checked entry points, a proof-carrying resolved function pointer, cached
selection, and chunked paths that expose dispatch-boundary costs. The Linux
runner retains correctness, build, host, source, generated-code, and
fresh-process timing evidence when a run completes.

## Measurement boundary

Each process creates and verifies its own deterministic 64 MiB input, then
times eight complete passes. The timed interval contains only the selected
dispatch path and byte-counting kernel. Twelve process pairs form each
comparison, with six pairs in each order. Inner passes are not independent
samples.

## Interpretation boundary

Elapsed times are measured. Linked instructions are observed. Summary
statistics and paired ratios are derived. The named microarchitectural causes
of a timing difference remain inferred. Results apply only to each recorded
host, binary, toolchain, flags, input, and run window.

## Outcome

Treat CPU discovery as a safety lattice. Put profitability in an explicit
policy that can consider input size and measured host behavior, and place the
dispatch boundary where its fixed costs are amortized.
