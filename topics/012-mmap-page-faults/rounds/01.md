# Round 1: separate mapping, residency, PTEs, and faults

Date: 2026-07-22

## Question

What do Linux minor and major fault counts establish about an `mmap` access,
and which costs remain outside those labels?

## Artifact

The C workload creates four controlled first-access states and emits setup time,
touch time, `getrusage()` deltas, `mincore()` residency, and a checksum. The
runner uses eight order-balanced blocks of fresh processes, binds execution to
the supplied archive SHA-256, retains host and generated-code evidence, and
rejects an unverified cold-file state.

The Rust crate encodes two interpretation rules:

- A cold label requires zero pages resident before access, full residency after
  access, and a nonzero major-fault count.
- Time divided by fault count is an amortized phase metric, not handler latency.

## Measurement boundary

`CLOCK_MONOTONIC_RAW` covers one volatile byte access per runtime base page.
File creation, `fdatasync`, mapping, advice, `mincore()`, process startup, and
teardown remain outside the touch interval. The CSV retains setup time so the
moved work does not disappear from the record.

Each host supplies eight process observations per mode. Summary dispersion
covers process-to-process variation. It does not treat pages or inner-loop
iterations as independent samples.

## Interpretation boundary

Linux man-pages describe a major fault as one requiring I/O, but upstream v6.18
also counts successfully completed retried faults as major. The process counter
has no address attribution. A minor fault can still allocate, zero, charge,
lock, reclaim, or construct page tables. Fault-around, readahead, folios, and
huge pages break a one-fault-per-page model.

The final host records will distinguish measured elapsed time and counters,
observed code generation and CPU identity, and inferred VM mechanisms.
